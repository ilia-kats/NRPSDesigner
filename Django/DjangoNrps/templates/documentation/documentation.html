NRPSDesigner Docs

Welcome to the documentation of the NRPSDesigner. Find a detailed description of all NRPSDesigner functionalities.

Features

  • Graphical NRP design platform
  • In silico prediction of the corresponding optimal NRPS domain sequence according to the evolutionary distance of the host organism, cloning practicability and experimentally characterized domain interactions
  • Framework for the development of a cloning strategy including Gibson primer design, integration of the NRPS sequence into a standard or custom backbone and the incorporation of additional biological parts
  • Automated extension of the NRPSDesigner database by newly discovered NRPS pathways and further characterization of existing domains upon experimental testing
  • Integration of Gibthon to facilitate implementation of cloning strategy
  • Parts registry interface and SBOL output format
  • RFC100 for non-ribosomal peptide synthesis design

    • Workflow

    In general, the steps for creation of novel non-ribosomal synthetases using the NRPSDesigner are the following.

  • The NRP has to be chosen (sequence of amino acids).
  • An optimal NRPS domain sequence is returned.
  • The NRPS DNA sequences can be inspected.
  • Appropriate backbone (with a promoter, RBS, start and stop codon) and other parts can be added to the plasmid.
  • Gibson assembly can be used in order to construct the plasmid and transform bacteria.
  • Experimental results can be entered into the database, thus further characterizing the corresponding domains.
  • optional A library is created by shuffling one of the monomer positions.
  • optional A library is created by shifting the boundaries of the domains.
  • optional The NRPSDesigner database can be extended with newly discovered synthetase sequences.

    • Novel NRPS

    The core functionality of the NRPSDesigner can be accessed by clicking on the "Apps" dropdown and then on the "NRPS Designer" button. Users that have not been logged in will be able to immediately proceed with the design of their synthetase of choice. Registered users will be redirected to their collection of already designed synthetases and thus will be able to reinspect or continue working on one of these or to design a new one.

    Input

    The input basically only consists of a sequence of monomers (amino acids) which should make up the final non-ribosomal peptide (NRP). This is achieved with the user-friendly interface shown below:


    Here, the actual sequence of NRP monomers can be stepwise selected. If a particular monomer is chosen its configuration can be specified as L or D. Note that the monomer choices available will depend on the specificities of the adenylation and condensation domains available in NRPSDesigner database and the chirality of the preceding monomer, as dictated by the rules of NRP synthesis.

    When you want to elongate the sequence of monomers by one, you should press the ‘Add monomer’ button. You can click on the chosen monomers and then possibly replace the selection by choosing another one from the dropdown. Finally, the peptide to be assembled is also presented graphically in the panel to the right.


    The NRPSDesigner makes it very easy to extend the database with novel NRPS genes and every user can do so (see below). Those genes and domains will also be used in the NRPSDesigner algorithm, if the above button has not been pressed. On the other hand, if it has been selected, then only domains curated by the authors, or other scientists with NRPS expertise (curators) will be utilized.


    Once all preferences have been set, the "Design construct" button can be pressed and a request will be sent to the NRPSDesigner database.

    Domain Output

    Once the request to the NRPSDesigner has been processed, the optimal NRPS domain sequence according to evolutionary distance, cloning practicability and experimentally characterized interactions is calculated by a rule-based algorithm (Dijkstra's algorithm). The visualization returned shows the domains which are necessary for the production of a synthetase capable of producing a peptide of choice.


    Plasmid view

    The NRPS domains shown in the previous visualization are encoded on the coding DNA sequences. Thus, as also exemplified in the screenshot below, the next step through the NRPSDesigner wizard leads to the actual DNA sequences and the fragments that have to be assembled.


    Note that each fragment shown in the plasmid map corresponds to a particular sequence that has to be amplified by PCR from a source organism. For example, the clicked fragment above corresponds to a DNA sequence that has to be amplified from Penicillium chrysogenum . Also observe that this fragment codes for more than one domain, namely a thiolation, an epimerisations and a thioesterase domain. In particular, the rules that produce the domain score, which is integrated in Dijkstra's algorithm also enforce the minimization of the number of fragments that have to be amplified. Thus, less cloning steps will be necessary and less native NRPS domain interfaces will be disrupted.

    Download SBOL Download GenBank

    The DNA sequences of all those fragments can be downloaded from this panel in both SBOL and GenBank format with the above buttons. They form the basic requirement for further use of the NRPSDesigner output, e.g. with available Bio-CAD tools.

    Create monomer library Create boundary library

    Also, it is possible to create combinatorial libraries by starting from this construct, either by shuffling one of the monomers or by changing the boundaries of some of the domains. This functionality, which can be accessed with the above buttons, will be explained in a subsequent section.

    Important Finally, note that synthetic NRPS are also functional when an adenylation domain, natively found in an elongation module, is used as the initiation domain. The consequence for the NRPSDesigner is that returned NRPS coding sequences may start without a start codon. Thus, in the subsequent steps of design of an appropriate cloning strategy, it is essential to introduce a start codon e.g. by including it in the backbone. Similarly, in order to enable more complicated experimental strategies, for example the addition of a GFP at the end of the synthetic NRPS, no Stop codon is added automatically, thus the stop codon also has to be introduced in the subsequent cloning strategy.

    Cloning strategy

    The previously mentioned steps comprise the core functionality of the NRPSDesigner: The selection of appropriate DNA sequences and their source, which after assembly and transformation will lead to the production of a non-ribosomal peptide of choice. Thus, the subsequent steps can range from direct synthesis to diverse cloning techniques, such as Golden Gate, CPEC or Gibson assembly.

    In fact, it is highly encouraged to use your own tool for CAD DNA design and assembly for the next steps, for example commercial programs such as teselagen's platform or J5 . Nevertheless, here we already provide an in-built solution for the CAD DNA assembly, namely Gibthon.

    About Gibthon

    Gibthon was started as the software project of the Cambridge iGEM teams of 2010 and 2011 and comprises an open source tool which aids "in the design and manufacture of synthetic parts and devices for biological systems." The manufacture of the synthetic parts is accomplished by means of Gibson assembly, a technique for one-pot isothermal and scarless assembly of DNA fragments. Gibthon was chosen because of its user-friendly interface, which rapidly allows the scientist to use his expertise and modify the Gibson assembly primer suggestions by having direct access to the sequences. After each sequence modification, important primer properties (e.g. regarding self-priming) are re-calculated using UNAfold. In the next sections, the adapted version of Gibthon and its integration in NRPSDesigner will be explained.

    The first step when using Gibthon is to add the appropriate DNA fragments that have to be assembled to the library. In the case of the NRPSDesigner, the coding sequence for each synthetase is automatically returned. Yet, you still need to add your own DNA parts, for example a backbone with a promoter and a ribosome binding site. Note that some biological devices of general use are available to all users from the Parts Registry .

This can be achieved from the "apps" dropdown and then choosing "Manage Gibson fragments". There, a user can access all the parts that he has already added (e.g. delete them or download their DNA sequence), as well as add new parts, as seen in the screenshot above. Of course it is also possible to add sequences by submitting FASTA/Genbank files or by copying the raw sequence string.

    Plasmid view #2

    The previously presented dynamic Gibthon plasmid view can be utilized for a Gibson assembly strategy. In particular, parts that have been added to the user specific Gibson fragment library (for registered users) or globally available parts (available to all users) can be added to the plasmid by simple drag and drop. Dragging towards the inner side of the plasmid results in the fragments being entered in the reverse direction (as seen in the screenshot below), else the sequence is added in the forward direction.


    The behaviour in the previously mentioned step is similar to the one present in the original Gibthon web-server, but with one important difference: Other fragments cannot be entered into the plasmid if they would disrupt the previously returned sequence of NRPS domains. Thus, other DNA sequences can be embedded in the plasmid only after the thioesterase (TE) domain or before the initialization adenylation domain.

    Important It is very important to ensure the addition of a promoter, a ribosome binding site, a start and a stop codon in the final sequence!

    Primers

    If you click on the "Primers" tab, you will initially come across the following loading bar:


    Clicking on "Design primers", will calculate appropriate primers, as seen on the next screenshot:


    In particular, the generated overview of the primers, includes a graphical display of the fragments to be assembled and the corresponding primers. For each of these, the visual representation shows to which fragment the annealing sequence of the primer binds to and to which fragment the Gibson overhang is complementary to. In addition, a small table summarizes all primers, their length and melting temperature (annealing end, whole primer). In the example above, 8 primers have been generated, corresponding to the 4 fragments (3 NRPS sequences, 1 backbone).

    Also note, that the graphical representation of the primers, can include a number (inside a red box). This numbers corresponds to the number of the following types of warnings:

  • Tm warning: the annealing temperature of either the annealing portion or the whole primer is less than the target
  • Self prime warning: the primer could potentially prime to itself on the 3' end
  • Misprime warning: the primer could potentially prime elsewhere on the fragment

    • The last two warnings are actually produced by UNAfold.

    For more information about a specific primer, one can click on it (in the graphical representation), as exemplified in the following screenshot:


    Note that the primer specific view includes the sequence of the primer as well as a graphical representation of the primer and the two corresponding DNA sequences: The DNA sequence which gets amplified with PCR due to the 3' sticky end of the primer as well as the sequence which will be combined with the aforementioned one during Gibson assembly. The view also includes a detailed description of all warnings and a boxplot generated by UNAfold (see next section). Clicking on a particular base pair of the DNA sequences on the graphical view will save the primer with the new length. Warnings and melting temperature will be re-calculated.


    Boxplot


    A sample self-priming boxplot is shown above. This is the output of the folding program used to determine whether any self annealing will occur. Each mark on the plot represents a possible basepairing. The color determines the likelihood of pair forming.

    Black represents the lowest free energy, followed by red, green, and yellow. The lower the free energy, the more likely the structure is to form. As a general rule of thumb, if it is positive, it will be fine.If it is greater than ~-3 kcal/mol, it will probably be OK. Any lower than this and, depending on the position, you will probably want to consider redesigning your primers.

    The right hand side of the plot corresponds to the 3' end of the primer. If there are any marks on this column, then the 3' end may be bound to another part of the primer. This will prevent extension of the primer along the target fragment. Note that the boxplot auto-scales to fit the provided data, so make sure you compare the number in the top right with the length of your primer.

    This boxplot can depict the formation of secondary structures. However, the most likely to form (at 22 bp from the 5' end) still has quite a high free energy, and due to its position is unlikely to affect the ability of the 3' end to extend. The original primer was 40bp long, so the secondary structure forming at 39bp from the 5' end does not include the 3' base pair. It is also a much higher free energy, and so is very unlikely to appear.

    Monomer libraries

    The NRPSDesigner also provides an experimental feature for high-throughput generation of peptide libraries by starting from a given scaffold (with applications for example in the case of production of novel antibiotics by starting from a known non-ribosomal peptide antibiotic). This functionality can be accessed, from the Designer tab of the main construct, as shown previously. Logged-in users can also access this functionality by visiting their collection of designed NRPs and then clicking on the downwards-pointing arrow of their peptide of choice and then choosing "Create monomer library", as seen in the screenshot below.


    In the library view, the sequence of monomers of the NRP produced by the scaffold is highlighted. The heuristic utilized here for high throughput is the following: For the creation of a library, one particular monomer position can be shuffled with many different other monomers. Thus, most of the primers for the rest of the scaffold can be reused and only the set of primers necessary for the domain to be shuffled has to be ordered. The corresponding interface can be seen in the screenshot below:


    The monomer position to be exchanged can be chosen by clicking on the corresponding or button. Choosing the option leads to the shuffling of that particular position with all monomers of the database. On the other hand, by repeated clicking of the button, a manual selection of monomers to be used for the exchange can be accomplished. Once preferences have been set, the appropriate constructs are designed by clicking on the "Design library" button. Note that depending on the number of monomers, this procedure could take a long time. Once the calculation is ready, a downloadable output with SBOL/GenBank files, as well as primers, similar to the one described for the scaffold construct, is returned.

    Boundary libraries

    Similarly to the monomer library functionality, a boundary library functionality exists, in which the user can modify the boundaries of the domains included in the original construct. For each pair of adjacent domains, the protein sequence of the corresponding synthetases which natively code for then, can be shown. This makes it easy to choose different boundaries for a particular domain combination, as exemplified in the screenshot of this functionality below:


    In this example, the user wants to modify the boundary/linker between an Adenylation and a Thiolation domain. In the left panel, the adenylation domain is highlighted in green. All other domains in this synthetase have a blue tooltip below them, such as the thiolation domain which follows the adenylation domain. Likewise, in the right panel, the thiolation domain is highlighted in green. The yellow amino acids correspond to the amino acids the user has chosen for the corresponding boundaries. In this case, the two domains will be linked at the highlighted P and A residues (i.e. the designed NRPS will include the sequence AARTP-AEIQLTA).

    This functionality enables experts to easily apply their knowledge regarding functional NRPS domain boundaries while designing their constructs and can also serve as a platform for elucidating this relationship.

    Extending the database

    [Login required]

    As mentioned previously, every user can extend the NRPSDesigner database by adding new NRPS pathways (with the safety mechanism of distinguishing between sequences entered by curators or normal users). This functionality can be accessed from the "Apps" dropdown by choosing the "Add NRPS into library" option. Exemplary, a sample input of the tycB gene is shown below:


    Beyond the source organism and the product synthesized in the native organism, the DNA sequence has to be entered. This can be a raw string or a FASTA formatted sequence. Here a number of conditions are checked: The DNA sequence entered has to be a coding sequence. Thus, its length has to be a multiple of 3 and it has to begin with a start codon, end with a stop codon and no stop codons should be interspersed within the gene (start/stop codon definition according to the NCBI Bacterial, Archaeal and Plant Plastid Code ). If one of these conditions is violated, appropriate warnings are returned.

    Once everything has been entered, the request can be submitted to the NRPSDesigner server by pressing the "Generate Domains" button. The pipeline based on hidden markov models (HMM) then detects all NRPS domains. If none is found, then an appropriate error message is returned. In the example of the tycB gene, 11 domains are detected and returned.


    Once this output has been returned, manual curation has to take place: This includes adding appropriate descriptions to the files, checking or adding specificities to the adenylation domains and checking domain boundaries. In particular, for each of the domains the following fields can be modified:

    • Module: The number of the module to which the domain belongs. This is automatically calculated, but can be readjusted in the case of several genes coding for the same synthetase (for example the module number of tycC domains could be readjusted to start at 5). In this case, the final domain of tycB interacts with the first domain of tycB during the production of Tyrocidine in nature. Thus, by tagging them with consequent module numbers, the NRPSDesigner scoring function can prioritize them accordingly.
    • Domain Type: The type of the domain which was predicted.
    • Substrate Specificity: In the case of adenylation domains, it is very important to check for the correct specificity. If the antiSMASH2 substrate specificity prediction reaches a consensus, then this consensus is displayed. Nevertheless, this prediction might be wrong, so adjust accordingly. Condensation domains also have a specificity for the acceptor substrate. The input of this type of substrate specificity is optional, as it can be inferred from the specificity of the subsequent adenylation domain.
    • Chirality: (Optional): The chirality of a condensation domain describes its specificity for the preceding monomer, which is based on the monomer's chirality. This does not have to be entered in general, as the HMMs can distinguish between "C_L" and "C_D" domains (condensation domains with L or D specificities respectively).
    • Predicted Start, Predicted Stop: The start and stop of the domain (left and right boundary), as predicted by the HMM pipeline. Cannot be changed. Once all domains have been submitted and entered into the database, you will be able to enter experimental evidence for other domain boundaries by the appropriate NRPSDesigner application ("Submit experimental data").
    • Description: An optional short description of some the domain's features or experimental findings regarding this domain (cite references which were added during the first step).

    Pressing on the "Save cds and domains!" button results in the storage of all the previously entered and predicted information to the NRPSDesigner database.

    Submit experimental data

    In this panel (see screenshot below) you can report the result of a successful NRPS assembly. If a construct has worked experimentally the domain borders of the respective database entries can be updated accordingly. First, you have to select two neighboring domains from your construct. Different filtering options help you to find the correct domains quickly in the database. Subsequently the corresponding coding sequence will be displayed. In green you can find the current domain length and borders. If your experimentally tested domain borders differ you can enter your result into the respective input fields e.g. by clicking on the corresponding base pair in the sequence shown. Finally, you can specify if your domain can be found also in external databases, as well as references validating this particular domain combination.


    Privacy

    Every user’s submitted data is private and not viewable by anyone other than the user. In particular, for non-registered users, safety is ensured by assigning a random UUID (according to RFC4122 ) to designed NRP constructs. In the case of registered users, both the identity of the user and the UUID of the constructs are used as safety checks. Also, the in-built authentication system encrypts the password of all registered users using PBKDF2 (Password-Based Key Derivation Function 2).
